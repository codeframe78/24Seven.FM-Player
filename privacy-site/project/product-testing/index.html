Prepare once

Make every result reproducible

Record the build and environment before touching the controls. Run only assigned account, posting, request, report, accessory, and future-delivery cases; mark a missing prerequisite Blocked rather than improvising around it.

  1. 1

    Identify the build

    Use an approved Play testing track or supplied release channel. Record the app version and code, installation source, device model, Android/API version, form factor, connection, theme, font/display scale, and relevant accessories.

  2. 2

    Take an assignment

    Record the PT case ID, station, and configuration. One report covers one case on one environment; repeat a case on another station or device only as assigned.

  3. 3

    Use safe test data

    Use only approved test accounts and synthetic dates or messages. Never disclose a real date of birth, tester identity, password, security answer, private conversation, or session value.

  4. 4

    Stop at the first unsafe boundary

    Do not send a Chat post, song request, abuse report, or external share unless that exact mutation is authorized. Do not repeat an indeterminate mutation; inspect the visible destination and report what happened.

Outcome vocabulary

Pass, notes, fail, or blocked

Pass means every observable boundary in the case occurred. Use Pass with notes when it worked but was confusing, slow, or inconsistent; Fail when an expected boundary did not occur; and Blocked when the build, account, permission, station, network control, device, or accessory was unavailable.

PT-01–PT-08 · Foundation and listening

Prove the app can arrive, play, and recover

These cases cover delivery, navigation, every station, the MediaSession, the service-owned timer, Android output routing, station data, and bounded network recovery.

PT-01 · Assigned install tester

Install, update, and first launch

  1. Install the assigned Play-delivered build or approved release APK and confirm the launcher icon, app label, version, and version code. On Android 13+, also enable themed icons and inspect the monochrome variant.
  2. Cold-launch once and exercise both notification-permission outcomes across the assigned Android 13+ devices; public playback must remain usable if permission is denied.
  3. Update an eligible earlier Alpha in place and verify local settings and protected station sessions remain available.
  4. On the designated clean-install device, uninstall and reinstall; confirm protected sessions and app-local choices are not restored from Android backup.
  5. Confirm launch has no crash, ANR, unexpected browser surface, or automatic audio start.

Pass: install and update complete under the expected identity; retained data survives only an in-place update, and a fresh install starts clean.

PT-02 · All configurations

Navigation, preferences, and exit

  1. Reach Player, Favorites, Chat, Queue, and More; confirm all five station cards are horizontally reachable.
  2. From each non-Player destination, use the persistent mini-player and return to the full Player.
  3. Choose station A, set Resume last station, select station B, then force-stop/relaunch; confirm station B opens selected without auto-playing.
  4. With station B selected, set Use current station at startup and record B as the fixed station. Select a different station C, then force-stop/relaunch; confirm station B—not the newer station C—opens selected without auto-playing.
  5. Exercise the two-Back exit flow. Verify both Keep listening and Stop and exit outcomes in separate runs.

Pass: navigation and selected state stay coherent, startup choices restore exactly, and exit never stops audio without an explicit choice.

PT-03 · Every station across the cohort

Foreground playback and station switching

  1. For the assigned station, start playback and observe Idle/Buffering/Playing or an understandable error.
  2. Confirm audible live audio, selected-station styling, title presentation, artwork behavior, format/bitrate when supplied, and matching mini-player/system metadata.
  3. Pause and resume, then move to the previous and next station, including wraparound at the ends.
  4. Switch stations while playing and confirm the prior stream and metadata do not continue as a second logical player.
  5. Stop, then start again deliberately. Repeat PT-03 until all five stations have a result.

Pass: controls and visible state agree, metadata is station-scoped, and only the selected station is audible.

PT-04 · System-media tester

Background playback and media controls

  1. Start playback, press Home, lock/unlock the device, and leave it playing for the assigned soak period.
  2. Use notification, lock-screen, Bluetooth/headset, and hardware play/pause controls where available.
  3. Tap the notification body and confirm the existing Player task opens instead of creating a duplicate task.
  4. Remove the app task from Recents while listening; verify the service/session and system media surface remain coherent.
  5. Exercise an assigned transient audio-focus interruption and a permanent interruption; verify resume occurs only when appropriate.
  6. Stop playback and confirm the system media state no longer implies that audio is active.

Pass: one MediaSession owns every control, background audio survives lifecycle changes, and foreground/system state never contradicts playback.

PT-05 · Sleep-timer tester

Sleep Timer lifecycle

  1. While playing, open Sleep Timer and exercise a preset plus valid custom 1- and 720-minute boundaries; verify invalid values cannot start a timer.
  2. Confirm the live countdown, then pause/resume, switch stations, background/lock the app, remove and reopen the task, and return; the original deadline continues.
  3. Adjust the active timer and verify it replaces the prior deadline.
  4. Cancel the timer and confirm playback continues; start another timer and use manual Stop to confirm both playback and timer clear.
  5. Run a one-minute locked-device expiry and confirm the existing player stops once, the timer clears, and any available system cancel action disappears.
  6. On the assigned reboot subcase, reboot with a timer active and confirm it fails safe instead of unexpectedly extending the deadline.

Pass: the service owns one bounded deadline that restores in the UI and never creates or controls a second player.

PT-06 · Accessory tester

Audio-output selection and route loss

  1. Confirm the Player's current output name agrees with the device actually producing media.
  2. Open Audio output, dismiss it, and confirm the same Player task remains.
  3. With assigned Bluetooth, wired/USB, hearing-aid, HDMI, automotive, or remote hardware, switch routes and verify Android System UI and the Player agree.
  4. Switch back to the device speaker, then disconnect the active accessory and observe the selected fallback route and playback state.
  5. If the system chooser is unavailable, confirm the app reports that boundary without breaking playback.

Pass: Android owns discovery/handoff, the displayed active route stays accurate, and route loss leaves a usable, truthful state. Google Cast is not claimed by this case.

PT-07 · Every station across the cohort

Queue, History, and station data

  1. Open Queue for the assigned station and verify ordered upcoming tracks plus recently played content.
  2. Check only supplied title, artist/composer, album, duration, artwork, requester, and request-message values appear; missing fields remain absent.
  3. Before community access is enabled, confirm public requester identity and messages remain hidden; after PT-14, confirm permitted attribution appears separately from track metadata.
  4. Refresh once, switch stations, and confirm old rows, artwork, requester identity, and error state do not leak into the new station.
  5. After one successful refresh, induce an assigned read failure and verify retained rows are visibly stale or failed rather than silently current.
  6. Cover SST/1980s/Adagio/Entranced extended lists and Death.FM's compact list across the cohort.

Pass: Queue and History remain bounded, ordered, capability-aware, station-scoped, and honest about unavailable data.

PT-08 · Network-control tester

Network loss, cancellation, and recovery

  1. Start live playback, disable the active connection, and wait through buffering/fallback to the explicit offline waiting state.
  2. Restore validated connectivity and verify playback prepares once and returns without another Play action.
  3. In separate runs, pause, stop, or switch stations while waiting; restoring connectivity must not restart the cancelled intent.
  4. Repeat one Wi-Fi/mobile-data handoff without a full outage.
  5. While offline, try Queue refresh, Chat refresh, account loading, Favorites, and library search. Confirm generic errors reveal no endpoint details; cached Queue data is marked stale and request actions fail closed.
  6. Restore service and confirm a deliberate refresh recovers the affected content.

Pass: recovery is bounded and intent-aware, cached state is labeled, mutations stay disabled without fresh evidence, and no private network detail reaches the UI.

PT-09–PT-19 · Accounts, participation, and support

Read-only portions may be run by any assigned tester. Posting, request, moderation-delivery, and cross-account mention checks require explicit approval and disposable test data.

  1. In More, confirm all five station account states are visible without changing the listening station.
  2. Load the assigned station's native sign-in challenge, refresh the security code once, and sign in with the approved account.

  5. Force-stop/relaunch again and verify the signed-out station remains cleared. If natural expiry occurs, record the explicit expired/recovery state.

Pass: credentials remain transient, protected sessions restore only for their originating station, and station-only logout remains isolated.

PT-10 · Approved account with Favorites

Favorites filtering, sorting, and availability

  1. Confirm the signed-out station-qualified gate, then sign in and load only the selected station's Favorite tracks.

  4. On the assigned large SST list, traverse from the first to the last of approximately 1,500 tracks and keep filtering/sorting responsive.
  5. Verify an available track shows green Request Now; queued/recent tracks show red Track Recently Played and cannot be selected; stale or ambiguous data fails closed.
  6. Switch station and sign out; confirm the list and account-derived state clear without leaking.

Pass: the server-owned list is complete, responsive, sortable, station-scoped, and never enables an unsafe request.

PT-11 · Read-only request browsing

Library search and pre-submit checks

  1. While signed out, browse the library and confirm request submission remains gated.
  2. Search useful assigned queries by Title, Album, Artist, and Genre; open results and verify album/track metadata.
  3. Exercise random and least-played suggestions, plus Library order and Play state sorting.
  4. Open an available track's confirmation, verify the exact station/account identity and track, then Cancel and confirm no request appears in Queue.
  5. Confirm Queue/recent restrictions, account state, membership, cooldown/readiness, request limits, stale/ambiguous data, and unsupported capabilities fail closed with accessible non-color semantics.
  6. Change one eligibility input before confirmation and verify the app refreshes sufficiently fresh station and Queue evidence instead of trusting the earlier row.
  7. With the confirmation open, switch station or sign out; verify Send is disabled or the request is cancelled, and no request is sent for the former station/account.

Pass: browsing works without mutation, Cancel is safe, and confirmation is reachable only after fresh, station-scoped eligibility.

PT-12 · Explicit mutation approval required

One-shot song request

  1. Confirm the coordinator-approved station/account is signed in, ready, outside cooldown, and allowed exactly one request.
  2. Select one eligible track and verify the confirmation identifies the intended station, account, and track.
  3. Before Confirm, verify fresh Queue/recent-play, membership, cooldown/readiness, request-limit, session, and station-identity checks still pass; assigned negative subcases must prevent submission.
  4. On SST, exercise the optional request message within its 80-character limit; on other stations confirm the SST-only field is absent.
  5. Confirm once and record the exact success, rejection, or indeterminate result. Never tap Confirm twice.
  6. Inspect Queue before any further action. Verify a received request appears once, with requester and optional message separate from metadata.
  7. If the result is indeterminate, do not retry. Confirm the same track is unavailable from both Library and Favorites until Queue evidence or a new station session safely resolves it. If a cooldown becomes visible, confirm it prevents another request.

Pass: one explicit confirmation causes at most one mutation, the UI reports uncertainty honestly, and no automatic or tester retry duplicates it.

Structured intake

Submit one case, station, and environment per result

The public form captures the PT case ID, build, device, station, outcome, affected areas, expected and actual behavior, repeatability, reproduction steps, and redacted evidence. A clean pass is evidence. Split unrelated failures or observations into separate reports.

Critical

Safety or data exposure

Credential/session exposure, unintended private content, destructive account behavior, or a repeatable crash that prevents use. Stop testing and report immediately without attaching sensitive material.

High / Medium

Core function failure

Playback, navigation, account isolation, requests, community safeguards, delivery, or an adaptive layout cannot complete its intended job. Record exact steps and repeatability.

Low / Note

Usability or polish

The task completes, but wording, spacing, latency, spoken feedback, or consistency should improve. Select Pass with notes when the expected result still occurred.
